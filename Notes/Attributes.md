# Terraform Attributes (`/conceptinside`)

## What is a Terraform Attribute?

A **Terraform attribute** is a **property (field) of a resource, data source, module, or variable** that stores a value.

Think of it like an object in programming:

```text
Person
 ├── Name
 ├── Age
 └── City
```

Here, **Name**, **Age**, and **City** are attributes.

Similarly, an Azure Virtual Machine has attributes such as:

```text
Azure VM
 ├── name
 ├── location
 ├── size
 ├── private_ip_address
 ├── public_ip_address
 └── id
```

Some attributes are **provided by you**, while others are **generated by Azure after the resource is created**. ([HashiCorp Developer][1])

---

# Two Types of Attributes

## 1. Input Attributes (Arguments)

These are values that **you define** in the Terraform configuration.

Example:

```hcl
resource "azurerm_resource_group" "rg" {
  name     = "rg-dev"
  location = "Central India"
}
```

Here,

| Attribute | Value         |
| --------- | ------------- |
| name      | rg-dev        |
| location  | Central India |

Terraform sends these values to Azure while creating the resource.

---

## 2. Output (Computed) Attributes

These are values **returned by the cloud provider** after resource creation.

Example:

```hcl
resource "azurerm_public_ip" "pip" {
  name                = "my-pip"
  location            = "Central India"
  resource_group_name = azurerm_resource_group.rg.name
  allocation_method   = "Static"
}
```

After `terraform apply`, Azure generates values like:

```text
id
ip_address
fqdn
resource_guid
```

You can access them as:

```hcl
azurerm_public_ip.pip.ip_address
```

Terraform doesn't know these values until the resource exists. ([HashiCorp Developer][1])

---

# Syntax

General syntax:

```hcl
resource_type.resource_name.attribute
```

Example:

```hcl
azurerm_resource_group.rg.location
```

Breakdown:

```text
azurerm_resource_group   → Resource Type

rg                       → Resource Name

location                 → Attribute
```

# Attribute Reference Between Resources

One resource can use another resource's attribute.

Example:

```hcl
resource "azurerm_resource_group" "rg" {

  name     = "Prod-RG"
  location = "Central India"

}

resource "azurerm_storage_account" "storage" {

  name                     = "devstorage123"
  resource_group_name      = azurerm_resource_group.rg.name
  location                 = azurerm_resource_group.rg.location

}
```

Flow:

```text
Resource Group
       │
       ├── name
       └── location
              │
              ▼
Storage Account
```

Terraform understands that the Storage Account depends on the Resource Group through these attribute references, creating an **implicit dependency**.

# Attribute vs Argument

This is a common interview topic.

| Argument                              | Attribute                                     |
| ------------------------------------- | --------------------------------------------- |
| Value you configure                   | Value you access                              |
| Used while creating the resource      | Available after or during evaluation          |
| Example: `location = "Central India"` | Example: `azurerm_resource_group.rg.location` |

In Terraform documentation, the configuration fields inside a block are usually called **arguments**, while the values exposed by resources (such as `id` or `public_ip`) are referred to as **attributes**. In casual usage, people often use the terms interchangeably. ([HashiCorp Developer][1])

# Accessing Attributes

## Dot Notation

```hcl
azurerm_resource_group.rg.name
```

---

## Index Notation (Lists)

```hcl
azurerm_network_interface.nic.ip_configuration[0].private_ip_address
```

---

## Map Attribute

```hcl
tags = {
  Environment = "Dev"
  Owner       = "Admin"
}
```

Access:

```hcl
azurerm_resource_group.rg.tags["Environment"]
```

Terraform supports dot notation for object attributes and bracket notation for list and map access. ([HashiCorp Developer][2])

# Interview Questions

### Q1. What is an attribute in Terraform?

**Answer:** An attribute is a property of a Terraform object. It can either be configured by the user (as an argument) or computed and exposed by the provider (such as `id` or `public_ip`). ([HashiCorp Developer][1])

---

### Q2. How do you access an attribute?

```hcl
resource_type.resource_name.attribute
```

Example:

```hcl
azurerm_resource_group.rg.id
```

---

### Q3. Can one resource use another resource's attributes?

**Yes.**

Example:

```hcl
resource_group_name = azurerm_resource_group.rg.name
```

This also creates an **implicit dependency**.

---

### Q4. Is `id` an input attribute?

**No.**

It is a **computed attribute** generated by Azure after the resource is created.

---

### Q5. Why are attributes important?

* Connect resources together
* Avoid hardcoding values
* Enable implicit dependencies
* Improve code reusability
* Make Terraform configurations more maintainable

---

# Key Takeaways

* **Attributes** are properties of Terraform resources.
* **Arguments** are values you set in configuration; **attributes** are values you read from resources.
* Use the syntax `resource_type.resource_name.attribute` to reference attributes.
* Referencing attributes between resources automatically creates **implicit dependencies**.
* Common computed attributes include `id`, `public_ip`, `private_ip`, and `primary_blob_endpoint`. ([HashiCorp Developer][1])
